## Part 1：主要な意思決定とその理由

### D-01：製品名は「Odim」
- 決定：北欧神話のオーディン（Odin）＋鴉フギン/ムニンに由来。
- 理由：知恵・全方位の観測・「世界を見る眼」という含意がプロダクト思想と一致。Huginn/Muninは機能名としても機能する。

### D-02：3D Globe と 2D Map の両方を採用
- 決定：マクロ＝3D Globe、ミクロ＝2D Map。ズーム階層で滑らかに切替。
- 理由：Globeは「資本の世界的流れ」を直感的に見せブランド体験を強くする。2Dは「特定地点の精密分析」に向く。Windy.comと同じ二刀流が最適。
- 却下案：Globeのみ → 国・用地レベルの精密作業に不向き。2Dのみ → ブランド体験とマクロ把握が弱い。

### D-03：初期ターゲットは米国のTier 1ヘッジファンド
- 決定：ヘッジファンド／クオンツファンドを初期顧客に。
- 理由：支払意欲が最も高く（年$500K-$2.5M）、POCが最速。中長期投資ユースケースとOdimのデータ更新頻度（日次）が一致。
- 却下：個人投資家 → 支払意欲が低く、Odimのシグナル粒度を活かす資本規模がない。

### D-04：データは自社スクレイピング、更新は日次バッチ
- 決定：公開データを自社スクレイピング。GitHub Actions日次cronで更新。
- 理由：コストゼロ。Odimは中長期判断ツールで、土地・建設データは日次更新で十分（ユーザー自身も「1日1回でよい」と明言）。

### D-05：課金は「基本料金 + API従量課金」
- 決定：ベース固定 + Huginn API使用量比例。
- 理由：2026年のAIプロダクトの主流。重課金ユーザーからコストを回収しやすい。

### D-06：事業化前のAIはGemini無料枠
- 決定：`gemini-2.5-flash`（安定既定）/ `gemini-2.5-flash-lite`（軽量）/ `gemini-flash-latest`（実験的な最新追随）。
- 理由：コストゼロ。Google公式は stable / preview / latest / experimental のモデル命名を定義しており、監査性が必要な推論ではstable model stringを使うべき。
- 移行設計：`lib/ai/provider.ts` で抽象化。事業化時に `AI_MODEL` を有料Geminiモデル / `claude` / `gpt` へ差し替え。

### D-07：Huginn/Muninは最初から実装
- 決定：MVPから組織別AI記憶を入れる。
- 理由：ユーザーの明示要望。かつOdimの差別化の核。後付けはアーキテクチャ的に困難。

### D-08：デザインは「精密機器（instrument）」の美学
- 決定：賑やかなSaaSでなく、観測所の計器のような静謐・精度・権威。
- 理由：顧客（ヘッジファンド・政府・防衛）の文化に合致。「真実を扱う道具」という思想の視覚化。AI-SaaSとの差別化。

### D-09：タグライン
- 決定：英語「No fakes. No noise. Just truth, and profit.」／短縮形「Seize truth. Seize profit.」
- 日本語原典：「フェイクもノイズもいらない、真実と利益を掴め」。

### D-10：書体はSpectral + IBM Plex Sans + IBM Plex Mono
- 決定：Interを使わない。Display=Spectral（知的セリフ）、UI=IBM Plex Sans、データ=IBM Plex Mono。
- 理由：Interは「AIデフォルト」の象徴。Spectralの知的なセリフがOdin/知恵の世界観を表現し、決定的な差別化になる。全て無料・OSS。

## Part 2：監査で発見・修正した論理的・技術的矛盾

> 本設計は「ミスの多い人間が作成した」と仮定し、冷徹に監査した。以下は発見した矛盾と修正。

### 矛盾 A：「グローバル launch」×「自社スクレイピング」×「コストゼロ」
- 問題：全世界の公開データ（FERC/PUC/建設許可を全世界分）を一度にスクレイピングするのは、単独開発者・コストゼロでは非現実的。
- 修正：Ontologyは国に依存しない汎用設計にする（実装済の設計方針）。プロダクトは「グローバル対応」だが、データ網羅は **米国優先 → 日本/台湾/EU → その他** の順で充填する。「グローバルにローンチ」の看板は維持しつつ、データ密度に地域差があることを正直に明示。

### 矛盾 B：「Kafkaでリアルタイム処理」
- 問題：ユーザーは「無料ならKafka」と述べたが、自前Kafkaは単独開発者には運用負荷が過大。かつOdimのユースケースは中長期判断で、ユーザー自身「日次更新でよい」と述べている。「リアルタイムKafka」と「日次でよい」は矛盾。
- 修正：Kafkaを不採用。**GitHub Actions日次バッチ**を採用。これが「無料・運用負荷最小・ユースケース適合」の三拍子で最適。

### 矛盾 C：「全機能を同時にローンチ」×「単独実装」×「完璧な商用品質」
- 問題：全機能を完璧な品質で文字通り「同時に」コーディングすることは物理的に不可能（実装には依存順がある）。
- 修正：「同時ローンチ」を「設計は一括・実装は依存順・公開は一括」と再定義（`08-ROADMAP.md`）。設計（このContext Folder）は既に全機能分が完成。実装は土台から積むが、中途半端な公開はせず、全機能が揃ってから一括公開する。ユーザーの意図（間引きローンチを避ける）は守られる。

### 矛盾 D：「衛星SARで建設進捗検知」×「コストゼロ」
- 問題：Planet Labs / ICEYE などの衛星SAR APIは有料。コストゼロと矛盾。
- 修正：無料枠では衛星機能の**UIは完成させる**が、データは「公開された低頻度衛星画像 + 建設許可テキスト」で代替。`config/sources.json` に有料衛星枠を後から追加できる構造にする（設定差し替え原則）。事業化時に有料衛星APIを足せばUIはそのまま機能する。

### 矛盾 E：Gemini無料枠のレート制限 vs 「本番商用品質」
- 問題：Gemini無料枠のRPM/RPD/TPMはモデル・プロジェクト・使用量ティアに依存し、Google AI Studioで確認する必要がある。複数組織が同時利用する本番商用には不足する可能性が高く、「商用リリース可能な状態」と矛盾しうる。
- 修正：「商用リリース可能な状態」を厳密に定義（`08-ROADMAP.md` Part 4）＝「無料枠で動く完全な商用品質プロダクトで、事業化＝APIキー差し替えのみ」。無料枠の間はAI Studioで確認した現在の実効上限に合わせ、クローズドβ（同時アクティブ組織数を絞る）として運用。これは設計の欠陥ではなく、意図された移行設計。

### 矛盾 F：Graph DB（Neo4j等）の導入
- 問題：当初案でNeo4j等のグラフDBを検討したが、単独開発者・コストゼロには運用過剰。別サービスの管理コストが発生。
- 修正：PostgreSQL（Supabase内）のリレーショナルモデル + 再帰CTEでOntologyグラフを表現。グラフDBは将来Apache AGE（PostgreSQL拡張）への移行余地を残す。これでサービス数を増やさず、コストゼロを維持。

### 矛盾 G：「ただのデータベースではない」×実装の現実
- 問題：ユーザーは「ただのDBにするな」と要求。しかし素朴に実装するとテーブル表示の集合体になりかねない。
- 修正：Odimの非DB性を3つの具体機能で担保する設計にした：①因果リンク（Ontology Links）、②推論層（SPVResolver/RDS/Triangulation）、③Huginnの因果探索。「データを見せる」のではなく「データを因果接続し判断に変える」。これが設計全体に反映されている。

### 矛盾 H：Narrative Layer の位置づけの曖昧さ
- 問題：「Narrativeはノイズ」と切り捨てる思想と、「Narrative Layerを持つ」設計が一見矛盾。
- 修正：Narrative Layer は明確に「**真実の源ではなく、検知トリガー兼・Reality乖離測定の対象**」と定義（`00-VISION.md` 哲学4、`03-ONTOLOGY.md`）。Narrativeシグナルを Ontology の確定リンクにしてはいけない、というルールを実装規則に明記。

### 矛盾 I：「個人投資家を狙う」というユーザーの当初想定
- 問題：ユーザーは当初ターゲットに個人投資家を含めたが、支払意欲・シグナル活用能力の点で不適合。
- 修正：個人投資家を初期ターゲットから除外（`01-PRODUCT-SPEC.md`）。理由を明示し、将来Tier制での部分開放の余地のみ残す。

## Part 3：未解決・将来判断が必要な事項

実装を進める中で、いずれユーザー（Souma）の判断が必要になる項目：

| 項目 | 内容 |
|---|---|
| 事業化タイミング | 無料枠 → 有料への移行をいつ行うか。クローズドβの規模をどこまで広げるか。 |

| オンプレ対応 | Tier 3（政府）向けのオンプレ版を作るか。作る場合の時期。 |
| MCP対応 | 外部AI AgentがOdimをMCP経由で叩く機能をいつ実装するか（設計はAPI-firstなので拡張容易）。 |
| リポジトリ公開可否 | GitHub Actionsを無制限で使うにはパブリックリポジトリが有利だが、コード公開の是非。 |

## Part 4：実装者が判断に迷ったときの原則

このファイルに答えがない判断に直面したら、以下の優先順位で決める：

1. **`00-VISION.md` の哲学に従う** — Reality > Narrative。断定しない。出典を持つ。
2. **コストゼロ原則を守る**（`02-ARCHITECTURE.md`）。
3. **設定差し替え原則を守る** — 将来の有料移行をコード書き換えなしにできる形にする。
4. **シンプルな方を選ぶ** — 単独開発者がClaude Code/CodeXで実装・保守できる複雑度に収める。
5. **それでも迷ったら、Soumaに質問する** — 推測で重要な構造を決めない。
