## 0. AIデザインと一流デザイナーの差（実装者が理解すべき本質）

リサーチで判明した「AIが作った感」の正体と、一流が違う点：

| AIデザインの病理 | 一流デザイナーがやること |
|---|---|
| **平均化** — 学習データの最頻パターンを出す。安全だが無個性。 | **視点(POV)を持つ** — 全ての判断が1つの思想に奉仕する。 |
| 階層がない — 全要素が同じ重さ。 | **残酷な階層付け** — 何を最初に見せ、何を隠すかを決める。 |
| 足し算ばかり — 装飾を盛る。 | **引き算** — 何を「足さないか」を知っている。 |
| トークンがない — ハードコードされた値。 | **システムで設計** — 色・余白・型をトークン化し一貫させる。 |
| 感情・文脈・ブランド物語の欠如。 | **物語を宿す** — ブランドの世界観を細部に込める。 |

### Odimが絶対に避ける「AIスロップ」банリスト（実装厳守）
以下は「AI製」を一発でバラす定番パターン。**1つでも使ったら失格**：
- ❌ 紫〜シアンのグラデーション（purple-cyan gradient）
- ❌ ガラスモーフィズム（半透明ぼかしカードの多用）
- ❌ 数値へのグラデーションテキスト
- ❌ 全く同一のカードを敷き詰めたグリッド
- ❌ バウンス／弾性のイージング（bounce/elastic easing）
- ❌ カードの中にカードを入れ子（nested cards）
- ❌ デフォルトTailwindの青(#3B82F6)・デフォルトの角丸・Interフォント
- ❌ 絵文字をUIの機能要素として使う（装飾としても原則禁止）
- ❌ 意味のない影の多用、ネオンカラー、過剰な彩度

## 1. デザイン思想（Odimの視点）

> **「Odimは、真実を映す精密機器（instrument）である。」**

Odimは賑やかなSaaSダッシュボードではない。観測所(observatory)の計器であり、北欧神話の知の神オーディンの眼である。
全てのデザイン判断は、次の3語に奉仕する：

1. **Precision（精度）** — 計器のような正確さ。曖昧さを排す。タイポも余白も数値も「測定された」感触。
2. **Calm（静謐）** — 情報は多いが、画面は静か。ノイズを足さない。ユーザーの認知に余白を残す。
3. **Authority（権威）** — 軽薄でない。「これは真実を扱う道具だ」と佇まいで伝える。

「楽しい」「かわいい」「親しみやすい」は**Odimの価値ではない**。Odimの美は、**静かな厳格さ**にある。

## 2. カラーシステム

### 2.1 哲学
- 背景は純黒(#000)でも、AIお決まりの#0d1117でもない。**「北の夜空」= 微かに冷たい深いインク**。
- アクセントは派手な黄ではなく、**「ルーン金(rune gold)」= 古い北欧の金細工の、彩度を抑えた金**。Odin/知恵の象徴。
- 7つのReality Layerの色は、ネオンを禁じ、**製図・科学計器のような落ち着いた色相**で統一する。

### 2.2 ベース（背景・サーフェス）

```css
--ink-900: #0B0D11;   /* アプリ最下層の背景。深いインク、微かに冷たい */
--ink-850: #0F1217;   /* 沈んだ領域（マップ背景など） */
--ink-800: #14171E;   /* 標準サーフェス／カード */
--ink-750: #1B1F28;   /* 浮いたサーフェス／hover状態 */
--ink-700: #242935;   /* 強い区切り／アクティブ背景 */
```

### 2.3 境界線（border）
影で領域を分けない。**細く低彩度の線**で分ける（計器のような端正さ）。

```css
--line-faint:  rgba(255,255,255,0.06);  /* 標準の区切り */
--line-soft:   rgba(255,255,255,0.10);  /* やや強い区切り */
--line-strong: rgba(255,255,255,0.16);  /* 入力欄・強調枠 */
```

### 2.4 テキスト（純白は使わない）

```css
--text-primary:    #ECEEF2;  /* 主要テキスト（ほぼ白だが眼に優しい） */
--text-secondary:  #9AA3B2;  /* 副次テキスト */
--text-tertiary:   #5D6675;  /* ラベル・補足 */
--text-quaternary: #3A4150;  /* 最も控えめ（プレースホルダ等） */
```

### 2.5 シグネチャーアクセント：ルーン金

```css
--rune:        #C9A961;  /* 主アクセント。古い金。Odimの「眼」 */
--rune-bright: #E0C078;  /* hover／active */
--rune-dim:    #8A7544;  /* 抑えたアクセント */
--rune-wash:   rgba(201,169,97,0.08);  /* アクセント背景の極薄塗り */
```

ルーン金は**希少に使う**。画面の主役ではなく、「ここに真実がある」と指す一点。乱用は厳禁。

### 2.6 7つのReality Layer カラー（統制されたセット）
全色を同程度の彩度・明度に揃え、どれも「叫ばない」。製図・科学計器の配色。

```css
--layer-energy:    #E0904A;  /* 電力：琥珀 */
--layer-cash:      #6FA86F;  /* 資本：抑えた緑（紙幣ではなくセージ） */
--layer-land:      #A78B5F;  /* 土地：大地のオーカー */
--layer-compute:   #5E8FB5;  /* 計算：鋼の青 */
--layer-water:     #4F9DA6;  /* 水：くすんだ青緑 */
--layer-material:  #9B6FA0;  /* 原材料：くすんだ藤 */
--layer-logistics: #8A93A3;  /* 物流：冷たい灰青（中立的な「移動」） */
```

### 2.7 セマンティックカラー（状態）
ネオンの赤緑は禁止。**抑えた**赤緑を使う。

```css
--positive: #5FA878;  /* 上昇・確認・成功 */
--negative: #C75D5D;  /* 下降・警戒（純赤#F00ではない） */
--critical: #D9694A;  /* 重大アラート：焦げた橙赤 */
--neutral:  #5D6675;  /* 中立 */

/* 信頼度バッジ */
--conf-high: #5FA878;
--conf-med:  #C9A961;
--conf-low:  #C75D5D;
```

### 2.8 グラデーションの扱い
- 原則使わない。使う場合は**同系色の微かな明度差のみ**（例: ink-800 → ink-850）。
- 異色相をまたぐグラデーション（特に紫-シアン）は**絶対禁止**。
- データ可視化のヒートマップは例外だが、その場合も単一色相の明度グラデーションに限る。

## 3. タイポグラフィ

### 3.1 書体選定（すべて無料・OSS）

Odimは3つの書体を役割で使い分ける。Interは使わない（AIデフォルトの象徴）。

| 役割 | 書体 | 理由 |
|---|---|---|
| **Display（見出し・ワードマーク）** | **Spectral**（Google Fonts） | 画面用に設計された知的なセリフ。Odin＝知恵・権威・「生成でなく著述された」佇まい。AI-SaaSとの決定的な差別化。 |
| **UI／本文** | **IBM Plex Sans**（Google Fonts） | データUI向けに設計。機関的な真剣さ。1/l/0/Oの判別が明快。 |
| **データ・コード・数値** | **IBM Plex Mono**（Google Fonts） | 全ての数値・シグナルコード・座標・タイムスタンプ・ティッカー。等幅で桁が揃う。 |

### 3.2 数値の絶対ルール
データプロダクトの生命線。**すべての数値に等幅数字を強制**：
```css
font-variant-numeric: tabular-nums;
```
表・カード・チャートの数値は IBM Plex Mono + tabular-nums。桁が縦に揃わないUIは「素人の証」。

### 3.3 タイプスケール
```css
--text-display:  44px / 1.1  / Spectral 600;       /* 大見出し（稀に） */
--text-h1:       28px / 1.2  / Spectral 600;       /* 画面タイトル */
--text-h2:       20px / 1.3  / IBM Plex Sans 600;  /* セクション見出し */
--text-h3:       15px / 1.4  / IBM Plex Sans 600;  /* カード見出し */
--text-body:     14px / 1.6  / IBM Plex Sans 400;  /* 本文 */
--text-small:    12px / 1.5  / IBM Plex Sans 400;  /* 補足 */
--text-mono:     12px / 1.5  / IBM Plex Mono 400;  /* データ */
--text-label:    10px / 1.4  / IBM Plex Mono 500;  /* ラベル（大文字＋字間広め） */
```

### 3.4 ラベルの様式
小さなラベル（カードのカテゴリ名等）は **IBM Plex Mono / 大文字 / letter-spacing: 0.12em**。
これがOdimの「計器パネル」感を作る重要な署名。

### 3.5 ワードマーク
- 表記: **ØDIM**（Oをスラッシュ付きのØに）。北欧文字の含意 + 視覚的署名。
- 書体: Spectral 600。字間 0.04em。
- 色: `--text-primary`。重要箇所のみ `--rune`。

## 4. スペーシング（余白）

### 4.1 スケール（4pxベース・幾何学的）
```css
--space-1:  4px;
--space-2:  8px;
--space-3:  12px;
--space-4:  16px;
--space-5:  24px;
--space-6:  32px;
--space-7:  48px;
--space-8:  64px;
```

### 4.2 余白の哲学
- **余白は機能**。詰め込まない。Odimの「静謐」は余白から生まれる。
- カード内パディングは最低 `--space-4`(16px)、できれば `--space-5`(24px)。
- セクション間は `--space-5`〜`--space-6`。
- 「情報密度が高い」と「窮屈」は違う。密度はあっても**呼吸する余白**を残す。

### 4.3 リズム
縦方向の間隔は必ずスケール上の値を使う。13px や 27px のような「中途半端な値」を禁止。
余白の一貫性こそが「設計された」感触を生む。

## 5. 角丸・線・影

### 5.1 角丸（控えめに）
```css
--radius-sm: 4px;   /* 小要素（バッジ・チップ） */
--radius-md: 6px;   /* 標準（カード・ボタン・入力欄） */
--radius-lg: 10px;  /* 大きな容器 */
```
過剰な角丸（16px+の「やわらかい」角）は禁止。Odimは計器であり、輪郭は端正でやや硬質。

### 5.2 線
領域分割は**影でなく線**で行う（`--line-faint`等）。計器パネルのような端正さ。

### 5.3 影
- 影は最小限。浮遊感の演出にのみ、ごく控えめに使う。
- 使う場合: `box-shadow: 0 8px 24px rgba(0,0,0,0.4);` 程度。
- 多重シャドウ・カラーシャドウ・ネオングロー禁止（データ可視化のglow効果はチャート内に限り許可）。

## 6. モーション（「ぬるっとした」動きの実装規定）

ユーザーの明確な要望：**画面遷移・パネルが滑らかで「ぬるっと」していること**。
これを実現するための厳密な規定。実装は **Motion（旧Framer Motion）**。

### 6.1 シグネチャー・イージング
Odimの全ての動きは、以下の「滑らかな減速」カーブを基本とする：

```css
--ease-odim:    cubic-bezier(0.22, 1, 0.36, 1);   /* 標準。滑らかに減速（ぬるっと） */
--ease-soft:    cubic-bezier(0.4, 0, 0.2, 1);     /* 補助。穏やかな緩急 */
--ease-out:     cubic-bezier(0.16, 1, 0.3, 1);    /* 入場用。最後がとても柔らかい */
```

❌ **バウンス・弾性イージング（`spring` の跳ね返り過多 / `back` / `elastic`）は禁止**（AIスロップ）。
動きは「跳ねる」のではなく「滑り込んで静止する」。

### 6.2 デュレーション
```css
--dur-micro:   140ms;   /* hover・色変化・小さなフィードバック */
--dur-fast:    260ms;   /* パネル開閉・要素の出入り */
--dur-base:    420ms;   /* 画面遷移・大きな移動 */
--dur-slow:    640ms;   /* Globe⇄2D Mapの切替など特別な遷移 */
```

### 6.3 画面遷移（8画面の切り替え）
- 旧画面: opacity 1→0 + わずかに下へ8px、`--dur-fast` / `--ease-odim`。
- 新画面: opacity 0→1 + 下8px→0、`--dur-base` / `--ease-out`、40msディレイ。
- クロスフェード気味に重ねる。**白いフラッシュや瞬間的な空白を作らない**。

### 6.4 パネル・モーダル
- スライドイン: 端から `--dur-fast` / `--ease-odim`。
- 中身は20msずつ stagger（時間差）でフェードイン。一気に出さず「滑り込ませる」。

### 6.5 Globe ⇄ 2D Map の遷移（最重要の「ぬるっと」体験）
- ズームインで地球儀が展開し2D地図へ。`--dur-slow` / `--ease-odim`。
- カメラ移動・スケール変化を連続的に補間。カクつき・瞬間切り替えを絶対に避ける。
- 遷移中もデータノードは保持し、フェードで受け渡す。

### 6.6 データの出現
- 数値カウンタは `--dur-base` で目標値へカウントアップ（ただし派手にしない）。
- リスト・テーブルの行は 24msずつ stagger フェードイン。
- ロード中はスケルトンを表示。スピナーの無限回転は避け、静かなパルスを使う。

### 6.7 hover・インタラクション
- hover は `--dur-micro`。色・境界線の変化のみ。要素を拡大(scale)しない（拡大は安っぽい）。
- クリック可能要素は、hoverで境界線が `--line-faint`→`--line-soft`、またはテキストが `--text-secondary`→`--text-primary`。

### 6.8 アクセシビリティ
`prefers-reduced-motion` を尊重。指定時はモーションを最小化（即時遷移）。

## 7. コンポーネント原則

### 7.1 カード
- 背景 `--ink-800`、境界 `--line-faint`、角丸 `--radius-md`。
- ヘッダ: `--text-label` 様式のラベル + 区切り線。
- **カードの入れ子禁止**。

### 7.2 ボタン
- Primary: 背景 `--rune`、文字 `--ink-900`、`--radius-md`。希少に使う。
- Secondary: 透明背景、境界 `--line-strong`、文字 `--text-secondary`。hoverで `--rune`。
- ボタン文字は IBM Plex Mono / 大文字 / 字間広め（計器感）。

### 7.3 表
- ヘッダ行は `--text-label` 様式。
- 数値列は右寄せ + IBM Plex Mono + tabular-nums。
- 行 hover で `--ink-750` の極薄背景。ゼブラ縞は使わない（線で十分）。

### 7.4 バッジ・チップ
- 角丸 `--radius-sm`、`--text-label` 様式。
- レイヤー色・信頼度色を背景に8%濃度の塗りで使う（`--rune-wash` の手法）。

### 7.5 データ可視化
- チャートの色は7レイヤーカラー / セマンティックカラーのみ。
- 軸・グリッド線は `--line-faint`。
- glow効果はチャート内のみ、ごく控えめに許可（マップのノード強調）。
- グラデーションは単一色相の明度差のみ。

## 8. アイコノグラフィ

- 線アイコン（stroke 1.5px）で統一。塗りアイコンと混在させない。
- 絵文字をUI機能要素に使わない。
- ライブラリは Lucide（OSS・無料）等。ただしアイコンは最小限——Odimはアイコンより**文字とデータ**で語る。

---

## 9. レイアウト原則

- グリッドベース。要素は必ずグリッドに乗せる。
- 「光学的整列」を意識：数値・ラベルの視覚的中心を揃える。
- 情報階層を3段で設計：①今すぐ見るべき ②文脈 ③詳細（隠してよい）。
- 1画面に主役は1つ。全要素を同じ重さにしない。

- [ ] 紫-シアングラデーション、ガラスモーフィズム、入れ子カード、バウンスを使っていないか？
- [ ] 色・余白・角丸は、すべてこのファイルのトークンから取っているか？（ハードコード禁止）
- [ ] 数値は IBM Plex Mono + tabular-nums か？
- [ ] フォントは Spectral / IBM Plex Sans / IBM Plex Mono のみか？（Inter禁止）
- [ ] 動きは `--ease-odim` 系で「滑り込んで静止」しているか？跳ねていないか？
- [ ] 余白はスケール上の値か？中途半端な値(13px等)はないか？
- [ ] 画面に「主役」が1つあり、階層が明確か？
- [ ] ルーン金を乱用していないか？（希少に、一点に）
- [ ] この画面は「精密機器」に見えるか？「賑やかなSaaS」に見えていないか？

> 最後の問いが最重要。Odimは計器である。静かで、正確で、権威がある。
> 「楽しい」「やわらかい」「賑やか」と感じたら、それは設計を誤っている。
